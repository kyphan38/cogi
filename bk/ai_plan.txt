# Thinking Training App - Phased Requirements

## Project Overview

**Goal:** A personal "thinking tool" (NOT a brain-training game) that helps improve real-world decision-making through structured exercises with AI-generated content.

**Core Philosophy:**
* Tool for decision support, not gamification
* AI provides alternative perspectives, not scores
* Metacognition (reflecting on your own thinking) is the primary value
* Real data > synthetic scenarios
* Confidence calibration > ELO scores

## Technical Decisions

| Decision | Choice |

| AI Provider | Claude (Haiku 3.5 default, Sonnet 4 for Generative + Weekly Review) |

## AI Model Routing Strategy

| Exercise Type | Model | Reason |
| :--- | :--- | :--- |
| Analytical | Claude Haiku 3.5 | Subtle issue embedding, nuanced fallacy tagging |
| Sequential | Claude Haiku 3.5 | Structured JSON, dependency logic |
| Systems | Claude Haiku 3.5 | Dependency graph, ripple effect analysis |
| Evaluative | Claude Haiku 3.5 | Trade-off analysis, matrix positioning |
| Generative | Claude Sonnet 4 | Debate partner needs real depth, not surface |
| Weekly Insight Review | Claude Sonnet 4 | Journal pattern analysis needs nuance |
| Phase 0 prototyping | Gemini 2.5 Flash | Free tier, fast iteration, swap to Claude later |

Estimated cost: ~$1.25/month at 60 exercises/month (48 Haiku + 12 Sonnet + 4 reviews)

## Why Claude over Gemini (for production)

1. **Embedded issue subtlety:** Claude writes naturally-sounding text with non-obvious fallacies. Gemini tends toward "textbook" examples that are too recognizable.
2. **Logical fallacy analysis:** Claude's Constitutional AI training gives it deeper understanding of rhetoric, argumentation, and cognitive biases.
3. **Debate partner quality:** Claude challenges arguments with genuine depth and doesn't soften pushback excessively. Critical for Generative Thinking mechanic.
4. **Gemini Flash kept for prototyping:** Generous free tier makes Phase 0 iteration cheap. Abstract AI layer allows seamless swap to Claude for production.

## Thinking Categories (Expert-grouped, no overlap)

Based on cognitive science, the 8 types you listed are consolidated into 5 distinct groups, each with a unique UI mechanic. This eliminates overlap while covering all skills:

**Group 1: Analytical Thinking (Detect & Deconstruct)**
What it trains: Finding hidden assumptions, biases, logical fallacies, weak evidence. UI Mechanic: Highlight & Tag Covers your: Phản biện + Phân tích

**Group 2: Sequential Thinking (Order & Dependency)**
What it trains: Cause-effect chains, prioritization, process sequencing. UI Mechanic: Sequence Drag & Drop (timeline ordering) Covers your: Logic + Giải quyết vấn đề

**Group 3: Systems Thinking (Connect & Impact)**
What it trains: Seeing ripple effects, dependencies, feedback loops, bottlenecks. UI Mechanic: Node-Connection (dependency graph drawing) Covers your: Hệ thống + Chiến lược

**Group 4: Evaluative Thinking (Weigh & Trade-off)**
What it trains: Comparing options under constraints, cost-benefit analysis, risk assessment. UI Mechanic: 2x2 Matrix Drop + Weighted scoring Covers your: Đánh đổi + Tối ưu hóa

**Group 5: Generative Thinking (Create & Reframe)**
What it trains: Producing original arguments, reframing problems, finding alternatives. UI Mechanic: Structured writing prompts + AI debate partner Covers your: Sáng tạo (and addresses the "recognition vs production" gap)

## Phase 1: MVP - Analytical Thinking (Highlight & Tag)

**Duration:** 1 Cursor session (plan + implement) **Goal:** One fully working exercise type with metacognition built in

**1.1 Exercise Setup Screen**
* [ ] Dashboard page with "Start Exercise" button
* [ ] Exercise setup: select thinking type (only "Analytical" available initially)
* [ ] Domain selector dropdown (pre-configured domains + custom)
* [ ] "Generate Exercise" button -> calls Gemini API

**1.2 Highlight & Tag Mechanic**
* [ ] Display AI-generated text passage (200-400 words)
* [ ] Text is selectable - user highlights segments with mouse
* [ ] Tag toolbar with color-coded tags:
    * **Logical Fallacy** (red)
    * **Hidden Assumption** (orange)
    * **Weak Evidence** (yellow)
    * **Bias / Motivated Reasoning** (purple)
    * **Valid Point** (green) - important: not everything is wrong
    * **Unclear / Needs More Info** (gray)
* [ ] User clicks highlighted text -> selects a tag -> tag appears as annotation
* [ ] User can remove/change tags
* [ ] Highlighted segments stored in state

**1.3 Confidence Calibration (CRITICAL - do before AI response)**
* [ ] Before submitting, slider appears: "How confident are you? (0-100%)"
* [ ] User must set confidence before seeing AI perspective
* [ ] Confidence score stored with exercise result

**1.3b Calibration - how `actualAccuracy` is computed (per thinking type)**

Define **before** shipping each type so confidence vs accuracy is never vague. Store `actualAccuracy` as 0–100 on `ConfidenceRecord` (same shape for all types; formula differs).

| Thinking type | What “correct” means | `actualAccuracy` (0–100) |
| :--- | :--- | :--- |
| **Analytical** | User highlights that match embedded issues (tag + span overlap) | Weighted: **70%** span overlap (IoU of character ranges vs each `embeddedIssues.textSegment`, best-match per issue) + **30%** tag correctness on matched spans. Partial credit if span overlaps ≥50% but tag wrong. Cap at 100. Decoys: no penalty for tagging as issue if user tags as “Valid” or skips. |
| **Sequential** | User ordering vs dependency DAG | **% of dependency constraints satisfied** (edge A→B respected in final order). Steps marked `isFlexible`: no score impact if order swapped within flexible group. |
| **Systems** | User edges vs `intendedConnections` | **% of intended edges** user drew with correct `(from, to, type)` triplet, capped by user’s edge budget (20). Shock event: optional +10% weighting if “affected” node set matches AI within defined tolerance (document exact rule in Phase 3 ticket). |
| **Evaluative** | Matrix / scoring vs AI intended | **Matrix:** quadrant match per option (binary per option, average). **Scoring:** mean absolute error vs AI `suggestedScores` per criterion, inverted to 0–100 scale (define normalization in Phase 4 ticket). |
| **Generative** | Not binary | **AI rubric score** 0–100 (Sonnet): clarity, counterargument strength, falsifiability - stored once per exercise; used only for calibration trend, not shown as “grade” to user in MVP unless in narrative form. |

Phase 1 implements **Analytical** row only in code; other rows are **spec locks** - when Phases 2–4 ship, implement exactly these definitions (adjust constants only if user-tested).

**1.4 AI Perspective Panel (NOT "scoring")**
* [ ] After submission, AI shows its analysis:
    * "Here's what I embedded in this text:" (list of intentional issues)
    * "Here's an additional perspective you might consider:" (things user may have missed)
    * "Your highlight of [X] is interesting - here's why that's debatable:"
* [ ] AI acknowledges valid points user found that weren't pre-planned
* [ ] No numerical score on the exercise itself
* [ ] Tone: collaborative peer, not judge

**1.5 Metacognition Journal (MANDATORY - cannot skip)**
* [ ] After viewing AI perspective, **3 prompts** are drawn from a **rotating prompt pool** (not the same 3 every time):
    * Maintain a pool of **≥12** short prompts covering: hesitation, assumptions, emotion, evidence, stakes, alternatives, time horizon, stakeholder blind spots (exact list in `lib/prompts/journal-pool.ts`).
    * **Rotation rule:** each exercise picks 3 prompts uniformly at random from prompts **not** used in the user’s **last 5** completed exercises (tracked in IndexedDB). If pool exhausted, reset exclusion window.
* [ ] **AI reference line (optional, one sentence):** before the three textareas, show a single gray line generated by the **same model as the exercise** (Haiku): e.g. “In your last journal about [domain], you noted [one paraphrased theme] - does that show up here?” Inputs: last **2–3** journal snippets for exercises in the **same domain** only (no cross-domain), max ~400 tokens. If no prior journals in domain, omit this line entirely (no API call).
* [ ] Minimum **2** prompts must have text (>10 characters) each before "Finish" (raises bar vs single-question gaming)
* [ ] Textarea input for each prompt
* [ ] "Continue" -> **1.6 Action Bridge** (cannot skip without filling required journal fields)

**1.6 Action Bridge (real-world transfer - MVP, not deferred)**
* [ ] After journal, one final required field (short text, ≥15 chars):
    * “Based on this exercise, what **ONE** concrete action will you take in real life?” (placeholder examples in UI)
* [ ] Stored on the exercise record (or linked `actionId`) and listed on the **dashboard** as open actions with exercise title + date
* [ ] **Weekly check-in (non-blocking card):** “Did you follow through on these actions?” yes/no per item; answers stored for calibration / weekly review input
* [ ] **Finish Exercise** (final control after Action Bridge) persists exercise + journal + action bridge atomically to IndexedDB
* [ ] Rationale: without this, practice stays inside the app; Action Bridge closes the loop immediately from the first exercise type

**1.7 Real Decision Log (loop outside the app)**
* [ ] Settings or dashboard: section **“Real decisions”** - user can add a row anytime: **decision text** (what they decided), **date**, **domain** (dropdown), optional **link** to a past exercise id
* [ ] Optional reminder: “Log the outcome in 7 days” (stored flag, not push notification)
* [ ] IndexedDB table `decisionLog`: `{ id, text, decidedAt, domain, linkedExerciseId | null, followUpNote, createdAt }`
* [ ] Weekly Review (Phase 5) may include **last 3** Real Decision Log entries as input for pattern surfacing (“You logged X but didn’t log outcomes - consider closing the loop”)
* [ ] Distinct from Action Bridge: Action Bridge is **per exercise**; Real Decision Log is **user-initiated** real life decisions (may be unrelated to any exercise)

**1.8 Data Schema (Dexie.js / IndexedDB)**

```typescript
interface Exercise {
  id: string;                 // UUID
  type: ThinkingType;         // 'analytical' | 'sequential' | etc.
  domain: string;
  generatedContent: string;   // The AI-generated text
  embeddedIssues: EmbeddedIssue[]; // What AI intentionally planted
  userHighlights: UserHighlight[];
  confidenceBefore: number;   // 0-100
  aiPerspective: string;      // AI's response after submission
  createdAt: Date;
  completedAt: Date | null;
}

interface UserHighlight {
  id: string;
  startOffset: number;
  endOffset: number;
  text: string;
  tag: TagType;
}

interface EmbeddedIssue {
  description: string;
  type: TagType;
  severity: 'obvious' | 'moderate' | 'subtle';
  textSegment: string;        // The part of text containing the issue
}

interface JournalEntry {
  id: string;
  exerciseId: string;
  promptIds: string[];        // which pool prompts were shown (for rotation)
  aiReferenceLine: string | null; // optional one-liner from Haiku
  responses: Record<string, string>; // promptId -> user text (min 2 entries required)
  createdAt: Date;
}

interface ActionBridge {
  id: string;
  exerciseId: string;
  oneAction: string;          // ≥15 chars
  weeklyFollowThrough: { weekKey: string; done: boolean }[];
  createdAt: Date;
}

interface RealDecisionLogEntry {
  id: string;
  text: string;
  decidedAt: Date;
  domain: string;
  linkedExerciseId: string | null;
  followUpNote: string | null;
  remindOutcomeAt: Date | null;
  createdAt: Date;
}

interface ConfidenceRecord {
  id: string;
  exerciseId: string;
  confidenceBefore: number;
  actualAccuracy: number;     // Calculated from matches
  gap: number;                // confidence - accuracy (overconfidence metric)
  createdAt: Date;
}

**1.9 Gemini Prompt Template (Analytical)**

SYSTEM: You are generating a thinking exercise. Return ONLY valid JSON.

USER context: {userContext}

Generate a {domain} analysis passage (250-350 words) that contains exactly:

1 obvious issue (most people would catch this)

2 moderate issues (requires careful reading)

1 subtle issue (only critical thinkers would catch)

2 "decoy" statements that LOOK suspicious but are actually valid

The passage should read naturally, like a real {domain} analysis or plan.
Do NOT make issues cartoonishly obvious.

Return JSON:

```JSON
{
  "title": "string - topic of the passage",
  "passage": "string - the full text",
  "embeddedIssues": [
    {
      "description": "what the issue is",
      "type": "logical_fallacy | hidden_assumption | weak_evidence | bias",
      "severity": "obvious | moderate | subtle",
      "textSegment": "exact substring from passage containing the issue",
      "explanation": "why this is an issue"
    }
  ],
  "validPoints": [
    {
      "textSegment": "exact substring that looks suspicious but is valid",
      "explanation": "why this is actually a valid point"
    }
  ]
}
```

**2.3 Gemini Prompt Template (Sequential)**

Generate a {domain} process-ordering exercise with 8 steps.

Requirements:
- Steps should have clear dependencies (A must happen before B)
- Include 2-3 steps where order is genuinely flexible
- Include 1 "trap" step that SEEMS like it should be first but actually depends on something else

Return JSON:

**6.1b Paste sanitization (security + model hygiene - before AI call)**

Apply **in the browser** immediately on paste / clipboard import, then again server-side on the received string in the API route (defense in depth):

* [ ] **Strip HTML:** parse as text-only (`DOMParser` + `textContent` or trusted serializer); reject or flatten if pasted content is HTML with `<script>`, `<iframe>`, `on*` event attributes, or `javascript:` URLs in anchors
* [ ] **Plain-text path:** prefer `navigator.clipboard.readText()` when available (no rich HTML)
* [ ] **Unicode normalize** to NFC; strip zero-width and bidi override characters that can spoof display
* [ ] **Control chars:** remove C0/C1 controls except `\n\t`; collapse runs of blank lines (max 2 consecutive)
* [ ] **Length:** enforce **2,000-word** cap after sanitization (same as existing guard)
* [ ] **Logging:** never log full pasted body server-side; at most hash + length for debugging

## Phase 7: Adaptive Difficulty (Post-validation)

**Duration:** 1 Cursor session **Goal:** Only build this AFTER you've used the app for 2+ weeks and confirmed value

**7.1 Difficulty Calibration**
* [ ] Track per-thinking-type performance metrics:
    * Analytical: % of issues correctly identified
    * Sequential: % of steps in correct dependency order
    * Systems: % of connections matching intended graph
    * Evaluative: distance from AI's matrix/scoring placement
    * Generative: AI assessment of argument strength (qualitative)
* [ ] AI adjusts prompt parameters based on performance:
    * Low performance -> fewer issues, more obvious ones, shorter text
    * High performance -> more subtle issues, longer text, more decoys
* [ ] Difficulty level shown as descriptive label, not number:
    * "Foundation", "Practitioner", "Advanced", "Expert"

**7.2 Weakness Queue (replaces SM-2)**

Design decision: SM-2 algorithm is designed for factual recall with clear right/wrong answers. Thinking skills don't have binary correctness - the same bias presented differently requires different recognition. SM-2's interval math doesn't apply here.

Weakness Queue is simpler and more appropriate:

* [ ] Schema migration - add new table to Dexie.js:

```typescript
interface WeaknessEntry {
  id: string;
  type: string;               // e.g., "confirmation_bias", "sunk_cost_fallacy"
  thinkingGroup: ThinkingType;
  missCount: number;          // times user failed to identify this
  hitCount: number;           // times user correctly identified this
  lastSeen: Date;
  status: 'active' | 'resolved';
  // resolved = hitCount >= 3 after becoming active
}

[ ] Detection: After each exercise, compare user's response vs AI's embedded issues. For each missed issue:

If weakness entry exists -> increment missCount

If not -> create new entry with missCount: 1

[ ] Injection: When generating new exercise, query active weaknesses. If any exist, append to AI prompt:

"Include at least one instance of [weakness type] in this exercise. Make it moderately difficult to detect."

[ ] Resolution: When user correctly identifies an active weakness type, increment hitCount. After hitCount >= 3 (since becoming active), set status to resolved.

[ ] Re-activation: If a resolved weakness is missed again in any future exercise, reset to active with hitCount = 0.

[ ] Dashboard display: Show top 3 active weaknesses on analytics page:

"Your current blind spots: [type] (missed X times, caught Y times)"

[ ] No intervals, no scheduling complexity. Weaknesses are simply injected into the next relevant exercise until resolved.

Acceptance Criteria:

Difficulty adjusts noticeably based on performance

Weakness Queue correctly tracks missed issue types

Active weaknesses appear in subsequent exercises

Resolved weaknesses can re-activate if missed again

No SM-2 interval scheduling - just simple queue logic
